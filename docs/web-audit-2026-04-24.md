# Auditoría web general — Somaz Studio

**Fecha:** 24 de abril de 2026  
**Alcance:** revisión técnica rápida basada en código + checks automáticos locales (lint, unit tests, e2e).

## Resumen ejecutivo

El sitio tiene una base sólida (Next.js 16, i18n, SEO técnico básico y suite de tests), pero hay oportunidades claras en **accesibilidad**, **higiene de QA**, y **robustez de CI**. Las mejoras más urgentes son:

1. Resolver errores de ESLint React Hooks (`setState` dentro de `useEffect`).
2. Corregir contrastes de texto por debajo de AA detectados por test.
3. Ajustar configuración de tests para no ejecutar suites de `node_modules`/carpetas ajenas.
4. Estabilizar E2E en CI (instalación de browsers Playwright y fallback de fuentes).

## Hallazgos

## 1) Calidad de código y rendimiento React

- Se detectaron errores de lint por `setState` síncrono dentro de `useEffect`, patrón que puede causar renders en cascada y degradación de performance percibida.
- También hay directivas `eslint-disable` sin uso, señal de deuda técnica menor.

**Impacto:** medio-alto (puede afectar fluidez de UX en componentes interactivos y mantenimiento del código).

**Recomendaciones:**
- Refactorizar para inicializar estado fuera de efecto cuando sea posible (initializer function en `useState`) o mover el cambio de estado a callbacks/subscripciones.
- Eliminar `eslint-disable` no utilizados para mantener el lint estricto y limpio.

## 2) Accesibilidad (WCAG)

- El test de contraste reporta clases de opacidad de texto (`text-foreground/30`, `/40`, `/45`) por debajo del umbral esperado.
- ESLint también reporta un `img` sin `alt` en generación de OpenGraph.

**Impacto:** alto (legibilidad, cumplimiento AA, SEO indirecto y experiencia de usuarios con baja visión).

**Recomendaciones:**
- Elevar opacidades mínimas en texto UI a valores compatibles con AA (según contraste real con fondo, idealmente >= 4.5:1 en texto normal).
- Añadir `alt=""` si la imagen OG es decorativa o texto alternativo significativo si aplica.
- Mantener y ampliar pruebas automáticas de contraste y axe en rutas críticas.

## 3) Fiabilidad de testing y CI

- `pnpm test` está ejecutando tests de una carpeta externa (`remotion/node_modules/...`), generando múltiples falsos fallos por dependencias de terceros faltantes.
- Los E2E fallan por falta de ejecutable de Chromium de Playwright en el entorno actual.

**Impacto:** alto (ruido en QA, pérdida de confianza en pipelines, señal falsa de regresiones).

**Recomendaciones:**
- Restringir `vitest` a `__tests__/**` y/o excluir explícitamente `node_modules`, `remotion/**`, `dist`, etc.
- En CI, ejecutar `pnpm exec playwright install --with-deps chromium` antes de `pnpm test:e2e`.
- Separar jobs: unit/lint (rápidos) vs e2e (más pesados) para feedback temprano.

## 4) SEO técnico y discoverability

- Existen `robots.ts` y `sitemap.ts`, lo cual es positivo.
- `sitemap.ts` usa fechas manuales por ruta estática, lo que puede quedar desactualizado sin un flujo de actualización.

**Impacto:** medio (frescura SEO y consistencia editorial).

**Recomendaciones:**
- Automatizar `lastModified` desde `git` o desde frontmatter/data source de contenido.
- Revisar trimestralmente prioridades (`priority`) y frecuencias de cambio según analytics de negocio.

## 5) Performance y resiliencia de assets

- Durante E2E aparecen advertencias de descarga de Google Fonts y fallback local.

**Impacto:** medio (variación visual/FOUT, potencial impacto en CWV según red/región).

**Recomendaciones:**
- Considerar self-hosting de fuentes clave o estrategia híbrida para reducir dependencia de red externa.
- Verificar preloads y pesos tipográficos realmente usados para reducir bytes iniciales.

## Plan sugerido (priorizado)

### Quick wins (1–2 días)
- Corregir errores de ESLint `set-state-in-effect`.
- Subir contraste de textos problemáticos.
- Limpiar directivas `eslint-disable` obsoletas.
- Ajustar `vitest` include/exclude para evitar suites externas.

### Sprint corto (3–5 días)
- Endurecer pipeline CI: lint + unit + e2e con instalación de browsers.
- Ejecutar baseline de Lighthouse/CrUX (mobile y desktop) y fijar objetivos (LCP, CLS, INP).
- Revisar componentes interactivos para accesibilidad teclado/focus visible.

### Mejora continua
- Auditoría trimestral de SEO on-page + datos estructurados.
- Presupuesto de performance por ruta (home, services, work/[slug], contact).
- Dashboards de errores de formulario/API y tasa de conversión por locale.

## Comandos ejecutados para esta auditoría

- `pnpm lint`
- `pnpm test`
- `pnpm test:e2e`

> Nota: esta auditoría es técnica y rápida; una auditoría completa debería sumar métricas de producción (Core Web Vitals reales, embudo, logs de errores, heatmaps y análisis de intención de búsqueda).
